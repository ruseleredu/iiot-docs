Node-RED vs ThingsBoard

Node-RED vs ThingsBoard​

Node-RED and ThingsBoard are both popular open-source tools in the IoT space, but they serve different primary purposes and are often used together rather than as direct competitors.

Overview​

AspectNode-REDThingsBoard
TypeLow-code flow-based programming toolFull-featured IoT platform (device management + visualization + processing)
Core StrengthIntegration, automation, data transformationDevice management, scalable data collection, dashboards, rule engine
DevelopedIBM (now OpenJS Foundation)ThingsBoard Inc. (open-source core)
ArchitectureNode.js-based, event-driven flowsJava/Spring, microservices support
DeploymentEdge (Raspberry Pi, gateways), servers, DockerCloud, on-prem, Edge, clustering for scale
LicenseApache 2.0 (fully open-source)Apache 2.0 CE + Professional Edition (paid)

Node-RED​

Node-RED is a browser-based flow editor for wiring together devices, APIs, and services. You create "flows" by dragging nodes (e.g., MQTT in/out, HTTP, function/JS, database, dashboard UI) and connecting them visually.

Key Features:

  • Thousands of community nodes (flows.nodered.org).
  • Excellent for protocol translation, data processing, automation.
  • Built-in dashboard for simple UIs.
  • Runs great on edge devices (very lightweight).
  • Strong in industrial integrations (Modbus, OPC-UA via nodes).
  • JavaScript flexibility inside Function nodes.

Pros:

  • Extremely flexible and extensible.
  • Fast prototyping and integration glue.
  • Low resource usage.
  • Active community and great for custom logic/automation.

Cons:

  • Dashboards are basic compared to dedicated platforms.
  • Not ideal as a standalone multi-tenant IoT backend (user management, scalability for millions of devices requires extra work).
  • Single-threaded nature (though you can scale with multiple instances).

Best for: Edge gateways, home automation, custom integrations, protocol bridging, simple automations.

ThingsBoard​

ThingsBoard is a dedicated open-source IoT platform for connecting devices, processing telemetry, managing assets/devices, setting up alarms, and building rich visualizations.

Key Features:

  • Device/asset management with digital twins and relations.
  • Powerful rule engine for data processing and alarms.
  • 600+ widgets for professional dashboards (real-time, SCADA-style).
  • Multi-tenancy (especially in PE).
  • Supports MQTT, HTTP, CoAP, and industrial protocols via Gateway.
  • Scalable clustering, fault-tolerant.
  • Mobile apps, white-labeling (PE), reporting, etc.

Pros:

  • Production-ready IoT backend out of the box.
  • Excellent data visualization and end-user dashboards.
  • Strong device management and scalability.
  • Community Edition is very capable.

Cons:

  • Steeper learning curve for complex rule chains.
  • Community Edition lacks some enterprise features (advanced RBAC, white-labeling, certain integrations — available in PE).
  • Heavier resource usage than Node-RED.

Best for: Full IoT solutions, monitoring platforms, SCADA-like apps, multi-customer deployments, large-scale data collection/visualization.

Direct Comparison​

  • Use Case Fit:

    • Need integration/automation/glue code on edge or server? → Node-RED wins.
    • Need robust backend, dashboards, device management, alarms? → ThingsBoard wins.
    • Need both (very common)? Use them together — Node-RED as gateway/edge processor sending data to ThingsBoard via MQTT.

  • Scalability: ThingsBoard has built-in clustering and is designed for large deployments. Node-RED can scale but requires more architecture work.

  • Visualization: ThingsBoard is far superior for professional, customizable, multi-user dashboards.

  • Ease of Use: Node-RED has a gentler curve for simple flows. ThingsBoard offers more "out-of-the-box" IoT functionality.

  • Cost: Both have strong free/open-source versions. ThingsBoard Professional Edition adds enterprise features and support.

Recommendation​

  • Start with Node-RED if you're doing prototyping, edge computing, or custom integrations.
  • Start with ThingsBoard if you need a complete IoT application platform with strong visualization and management.
  • Best combo (widely used): Run Node-RED on gateways/edge for local logic and protocol handling → forward clean data to ThingsBoard for storage, visualization, alarms, and multi-user access.

Many industrial and smart city projects use exactly this combination.

Node-RED and ThingsBoard integrate very well, with MQTT as the primary and most robust method. They are commonly paired in production: Node-RED handles edge data collection, protocol conversion, local processing, and automation, while ThingsBoard manages device lifecycle, scalable storage, advanced rule engine logic, alarms, and professional dashboards.

1. Direct MQTT Device API (Simplest – Single or Few Devices)​

Node-RED acts as a client publishing directly to ThingsBoard as one or more devices.

  • MQTT Broker settings in Node-RED:

    • Host: Your ThingsBoard instance (e.g., thingsboard.cloud or self-hosted IP).
    • Port: 1883 (plain) or 8883 (TLS).
    • Username: Device Access Token (most common) or MQTT Basic credentials.
    • Password: Usually empty for Access Token.

  • Common Topics (standard format):

    • Telemetry: v1/devices/me/telemetry
    • Attributes: v1/devices/me/attributes
    • RPC requests (from TB): Subscribe to v1/devices/me/rpc/request/+
    • RPC response: v1/devices/me/rpc/response/$request_id

  • Payload: Simple JSON, e.g., {"temperature": 23.5, "humidity": 45}.

Use case: Direct-connected sensors or simple flows. For multiple devices, you can use multiple MQTT Out nodes or dynamic connections.

2. Gateway MQTT API (Most Popular Pattern – Recommended for Edge)​

Node-RED acts as a ThingsBoard Gateway device. It uses one MQTT connection to proxy data for many downstream devices/assets. Perfect for industrial gateways, protocol translation (Modbus, OPC-UA, BACnet, etc.), and local networks.

Key Topics (all under v1/gateway/):

  • Telemetry: v1/gateway/telemetry
  • Attributes: v1/gateway/attributes
  • RPC: v1/gateway/rpc
  • Claiming, etc.

Payload Example (for telemetry):

{
  "DeviceA": [
    { "ts": 1720000000000, "values": { "temp": 25.3, "status": "running" } }
  ],
  "DeviceB": [{ "values": { "pressure": 101.2 } }]
}

Advantages:

  • Single connection to ThingsBoard.
  • Dynamic device creation (if enabled).
  • Better for 10s–1000s of devices behind the gateway.

There is a community node node-red-contrib-thingsboard-gateway that simplifies this.

3. Hybrid / Edge Processing Pattern​

This is the most common real-world architecture:

  1. Node-RED on Edge (Raspberry Pi, industrial gateways like Advantech ADAM, ARMxy, etc.):

    • Collect from PLCs, sensors, OPC-UA, Modbus, serial, etc.
    • Local filtering, aggregation, buffering, local logic/automation.
    • Protocol translation.
    • Publish clean data via Gateway API.

  2. ThingsBoard (Cloud or On-Prem):

    • Receives data.
    • Rule Engine for alarms, notifications, analytics.
    • Rich dashboards, widgets, mobile apps.
    • User/tenant management.

Bidirectional Control: ThingsBoard can send RPC commands (e.g., "turn on motor") via MQTT back to Node-RED, which executes them locally (e.g., via OPC-UA write).

4. REST API Integration​

Use the community node node-red-contrib-thingsboard-rest-api for:

  • Device/asset management.
  • Creating/updating entities.
  • Querying telemetry/attributes.
  • Dashboard operations.
  • When MQTT is not suitable (e.g., one-off admin tasks).

5. ThingsBoard Integrations (Reverse Direction)​

If devices talk to an external MQTT broker that Node-RED bridges or manages, use ThingsBoard's MQTT Integration (with uplink/downlink converters) to pull data in. Less common when Node-RED is the primary gateway.

Best Practices & Tips​

  • Security: Use TLS + strong tokens. For production, consider MQTT Basic or X.509.
  • Scalability: For many devices, prefer Gateway API + one stable connection. Use Node-RED subflows or dynamic MQTT nodes for large-scale handling.
  • Error Handling: Add debug nodes, catch nodes, and retry logic in Node-RED. Use ThingsBoard debug mode for integrations.
  • Local Resilience: Node-RED can buffer data locally (e.g., to SQLite/File) during connectivity loss.
  • Versioning: Keep flows modular (subflows, tabs). Use Git for version control of flows.
  • Industrial Examples: OPC-UA → Node-RED → ThingsBoard is very common for factory automation.

Example Resources​

  • Official ThingsBoard MQTT API docs.
  • Advantech ADAM + Node-RED guides (official device library).
  • Community flows on flows.nodered.org and GitHub examples.

Node-RED subflows are an excellent way to create reusable, maintainable gateway patterns for ThingsBoard integrations. They help encapsulate common logic (data formatting, protocol handling, error management, buffering) so you can deploy consistent behavior across many devices or data sources without duplicating flows.

Why Use Subflows for Gateway Patterns?​

  • Reusability — One subflow instance per device type or protocol.
  • Modularity — Clean main flows; complex logic hidden inside.
  • Maintainability — Update the subflow once, and all instances benefit.
  • Parameters — Expose configurable properties (device name, topic type, etc.) per instance.
  • Scalability — Easier to handle dozens or hundreds of sensors/PLCs.

Common Gateway Subflow Patterns​

1. Basic Telemetry Formatter (Most Common)​

This subflow takes raw data and formats it for the ThingsBoard Gateway MQTT API (v1/gateway/telemetry).

Recommended Inputs (msg properties):

  • msg.device or msg.payload.device — Device name in ThingsBoard.
  • msg.payload — Telemetry object (e.g., {temperature: 23.5, humidity: 45}).
  • Optional: msg.ts for timestamp.

Subflow Structure (typical nodes inside):

  • Function node: Format payload as { "DeviceName": [ { "ts": ..., "values": { ... } } ] }
  • Change node: Set msg.topic = "v1/gateway/telemetry"
  • Debug or Status node for visibility.
  • Optional: Switch for validation, Delay for rate limiting, or Catch for errors.

Example Function Code (inside subflow):

// Assume input: msg.device and msg.payload = telemetry object
const device = msg.device || msg.payload.device || "UnnamedDevice";
const values = msg.payload.values || msg.payload;

const telemetry = {};
telemetry[device] = [
  {
    ts: msg.ts || Date.now(),
    values: values,
  },
];

msg.topic = "v1/gateway/telemetry";
msg.payload = telemetry;
return msg;

Make device a Subflow Property (environment variable) so each instance can have its own device name.

2. Protocol-Specific Gateways (e.g., Modbus → TB)​

Create one subflow per protocol:

  • Modbus Reader + Formatter subflow.
  • OPC-UA Browser/Reader + Formatter.
  • Serial/ BACnet / etc.

Inside the subflow:

  • Protocol node(s) → Data processing → Telemetry formatter → MQTT Out (or output port).

Use Link Out or subflow output to a shared MQTT Out node for a single connection.

3. Bidirectional RPC Handler Subflow​

Handles commands from ThingsBoard back to edge devices.

  • MQTT In (subscribe to v1/gateway/rpc or device-specific).
  • Parse request.
  • Execute local action (e.g., write to PLC).
  • Send response back.

4. Buffered / Resilient Gateway Subflow​

  • Local storage (SQLite, File, or Context) during offline periods.
  • Batch sending.
  • QoS and retry logic.

Best Practices for Gateway Subflows​

  • Single MQTT Connection — Place the MQTT Out node outside the subflow (shared). Use subflow outputs to feed it. This avoids multiple connections to ThingsBoard.
  • Properties — Expose: Device Name, Topic Type (telemetry/attributes), Rate Limit, Enable RPC, etc.
  • Status Updates — Add a Status node inside the subflow to show connection/health on the instance node.
  • Error Handling — Use Catch nodes inside subflows and route to a central error handler.
  • Versioning — Add a description and version in subflow properties.
  • Testing — Use Inject nodes inside the subflow tab for easy testing.
  • Performance — Keep subflows lightweight; heavy computation can go in separate flows.

Advanced Tips​

  • Combine with node-red-contrib-thingsboard-gateway if available — it provides a dedicated node for gateway messaging.
  • Use Environment Variables or global config for ThingsBoard broker settings.
  • For very large deployments: Dynamic device handling + one subflow instance per logical group.
  • Modular design: "Collector Subflow" → "Processor Subflow" → "Publisher Subflow".

How to Create One​

  1. Select nodes → Menu → Subflows → Selection to Subflow.
  2. Double-click the subflow tab to edit.
  3. Add Input / Output nodes.
  4. Define Properties (Environment Variables).
  5. Deploy and drag the new subflow node from the palette.

Node-RED Modbus Gateway to ThingsBoard Example

This is a practical, production-oriented setup where Node-RED acts as an edge gateway:

  • Reads data from Modbus TCP (or RTU) devices/PLCs.
  • Uses a Subflow for clean, reusable formatting.
  • Sends data to ThingsBoard using the Gateway MQTT API (v1/gateway/telemetry).

Prerequisites​

  1. Install required nodes (via Manage Palette):

    • node-red-contrib-modbus (most popular and feature-rich).
    • Optional but recommended: node-red-contrib-thingsboard-gateway (simplifies TB gateway messages).

  2. Configure an MQTT broker connection to ThingsBoard (as a Gateway Device):

    • Create a device in ThingsBoard of type Gateway.
    • Use its Access Token as the MQTT username.

Recommended Architecture​

Modbus Reader → Processor (optional) → Telemetry Subflow → Shared MQTT Out

1. Create the Telemetry Formatter Subflow​

Name: TB Gateway Telemetry

Subflow Properties (Environment Variables):

  • deviceName (string) — e.g., PLC-01 or ModbusDevice1

Nodes inside the subflow:

  • Input node
  • Function node (core logic)
  • Change node (optional)
  • Output node
  • Status node (for visual feedback)

Function node code (inside subflow):

// Input expectations:
// msg.payload = raw value or object from Modbus
// msg.topic or msg.registerName = descriptive name (e.g., "temperature", "status")

const device = env.get("deviceName") || "UnnamedDevice";
const values = {};

// Handle different input formats
if (typeof msg.payload === "object") {
  Object.assign(values, msg.payload);
} else {
  const key = msg.registerName || msg.topic || "value";
  values[key] = msg.payload;
}

const telemetry = {};
telemetry[device] = [
  {
    ts: Date.now(),
    values: values,
  },
];

msg.topic = "v1/gateway/telemetry";
msg.payload = telemetry;

node.status({ text: device + " → TB", fill: "green", shape: "dot" });

return msg;

How it connects:

  • poll-inject → modbus-read
  • modbus-read → parse-registers (Function)
  • parse-registers → tb-telemetry-subflow
  • tb-telemetry-subflow → mqtt-out (shared connection)

3. MQTT Broker Configuration (tb-broker)​

  • Server: Your ThingsBoard host (e.g., thingsboard.mycompany.com or IP)
  • Port: 1883 or 8883 (TLS)
  • Client ID: Leave blank or unique
  • Username: Gateway Device Access Token
  • Password: (empty for token auth)

Advanced Options​

  • Multiple Devices: Create one subflow instance per device with different deviceName.
  • Batch Reading: Use modbus-flex-getter for more efficient polling of many registers.
  • RPC (Commands from ThingsBoard): Add an MQTT In node listening to v1/gateway/rpc and route commands to modbus-write nodes.
  • Offline Buffering: Add a catch node + SQLite/File storage + retry logic.
  • Error Handling: Use modbus-response node and status checks.

Tips for Production​

  • Use one shared MQTT Out node for all subflows (single connection to ThingsBoard).
  • Enable dynamic device creation in ThingsBoard (under Gateway settings).
  • Add rate limiting or deadband filtering to reduce unnecessary telemetry.
  • For Modbus RTU (serial): Change client type to serial and configure port/baudrate.
  • Monitor with Node-RED's Status nodes and ThingsBoard device activity.

OPC UA (Open Platform Communications Unified Architecture) is a key industrial protocol for secure, reliable data exchange with PLCs, machines, and SCADA systems. Node-RED integrates exceptionally well with OPC UA, making it a strong gateway to ThingsBoard for telemetry, monitoring, and bidirectional control.

Common Integration Patterns with ThingsBoard​

1. Polling Pattern (Simple & Reliable)​

  • Periodic read of multiple nodes/tags.
  • Map values → format for ThingsBoard Gateway API.
  • Send via shared MQTT Out node.

Typical Flow: Inject (timer) → OPC UA Item(s) → OPC UA Client (Read) → Function (map & format) → TB Telemetry Subflow → MQTT Out

2. Subscription Pattern (Event-Driven, Efficient)​

  • Subscribe to monitored items with deadband filtering.
  • Data arrives on change → process → send to ThingsBoard.
  • Lower network load than polling.

3. Bidirectional Control (RPC)​

  • ThingsBoard sends commands via MQTT (Gateway RPC).
  • Node-RED receives → OPC UA Write or Method Call → PLC.
  • Respond back to ThingsBoard.

4. Hybrid / Resilient Gateway​

  • Local buffering (SQLite) during outages.
  • Deadband + rate limiting to reduce telemetry volume.
  • Asset/relation modeling in ThingsBoard.

Example: OPC UA → ThingsBoard Gateway Subflow​

Subflow Name: OPCUA to TB Gateway

Properties:

  • deviceName (e.g., Machine-01)
  • opcEndpoint (optional for dynamic)

Inside the subflow (example logic):

// msg.payload from OPC UA Client is usually an object or array
let values = {};

if (Array.isArray(msg.payload)) {
  msg.payload.forEach((item) => {
    values[item.browseName || item.nodeId] = item.value;
  });
} else {
  values = msg.payload;
}

const telemetry = {};
telemetry[env.get("deviceName")] = [
  {
    ts: Date.now(),
    values: values,
  },
];

msg.topic = "v1/gateway/telemetry";
msg.payload = telemetry;

node.status({ text: "Sent", fill: "green", shape: "dot" });

return msg;

Full Basic Flow Sketch (Polling)​

  1. Inject node (every 5–10s).
  2. OPC UA Item nodes (define NodeIds, e.g., ns=2;s=Temperature).
  3. OPC UA Client (Read action, configured endpoint).
  4. Function or Join (combine multiple reads).
  5. TB Gateway Telemetry Subflow (from previous conversation).
  6. MQTT Out (shared connection to ThingsBoard Gateway device).

For subscriptions: Use the same Client node configured for Subscribe, with deadband.

Security & Production Tips​

  • Security Modes: Start with None/SignAndEncrypt + appropriate policies.
  • Certificates: Configure PKI folder in Node-RED for trusted certs.
  • Endpoint: opc.tcp://plc-ip:4840.
  • Performance: Use one shared OPC UA Client/Endpoint where possible. Batch reads.
  • Error Handling: Catch nodes + reconnect logic (many OPC UA nodes support auto-reconnect).
  • ThingsBoard Side: Enable dynamic device creation under the Gateway device. Create device profiles with attributes/telemetry keys matching your OPC UA tags.

Real-World Examples & Resources​

  • YouTube tutorials show Factory IO + Node-RED OPC UA + ThingsBoard motor control (read + write via RPC).
  • Community flows on flows.nodered.org and GitHub (search "node-red thingsboard opcua").
  • Browse examples in the OPC UA node palette (many built-in samples).
