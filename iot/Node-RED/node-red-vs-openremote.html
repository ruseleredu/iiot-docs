Node-RED vs OpenRemote

Node-RED and OpenRemote are both open-source tools in the IoT/automation space, but they serve different primary purposes and scales.

Overview​

Node-RED is a lightweight, flow-based, low-code programming tool for wiring together devices, APIs, services, and logic. It was originally developed by IBM and is now part of the OpenJS Foundation. It's extremely popular for rapid prototyping, home automation, and edge/IIoT integrations.

OpenRemote is a full-featured, 100% open-source IoT device management platform designed for professional integrators, OEMs, and larger-scale deployments (smart buildings, energy management, smart cities, fleet management, etc.). It provides a complete backend with asset modeling, multi-tenancy, rules, dashboards, and more.

Key Comparison​

AspectNode-REDOpenRemote
Core StrengthVisual flow programming & integrationsFull IoT platform (asset mgmt + automation + UI)
ArchitectureNode.js runtime, browser-based editorJava-based backend (Manager), with Flow editor, multi-tenant, scalable DB (PostgreSQL/TimescaleDB)
Asset ManagementNone built-in (you build it)Strong: Custom asset types, attributes, hierarchies, digital twins
Rules/AutomationExcellent visual flows + JavaScript functions. Huge community nodesWhen-Then rules, visual Flow Editor, Groovy scripts. Good for both simple and complex logic
VisualizationDashboard nodes (e.g., ui-dashboard)Built-in dashboard builder, maps/geofencing, Insights
ScalabilityGreat for edge/single instances. Can be clustered (with FlowFuse etc.) but requires more workDesigned for scale: Multi-realm (tenancy), edge gateways, OTA updates, high-volume time-series data
ProtocolsMassive community nodes (MQTT, Modbus, OPC-UA, HTTP, etc.)Built-in agents (MQTT, Modbus, BACnet, KNX, LoRaWAN, etc.) + extensible
Multi-tenancy / UsersBasic (add-ons needed)Strong (realms, Keycloak-based RBAC)
DeploymentVery easy on Raspberry Pi, Docker, etc.Docker-based, supports central + edge gateways
Use CasesHome automation, quick IIoT prototypes, data pipelines, glue logicProfessional solutions: Building/energy management, fleet, smart cities, OEM white-label products
Learning CurveVery gentle for flowsSteeper initially due to asset model & platform concepts
ExtensibilityExtremely high (thousands of npm nodes)Good (custom agents, web components for apps, APIs)
Community / MaturityVery large and activeSmaller but growing, focused on professional users

When to Choose Which?​

Choose Node-RED if:

  • You need quick integrations and automations.
  • You're doing home automation, prototyping, or edge processing.
  • You want maximum flexibility with community nodes and JavaScript.
  • You're comfortable building the higher-level parts (UI, user management, persistence) yourself or with add-ons.
  • Budget/time for a lightweight solution.

Choose OpenRemote if:

  • You need a production-grade IoT platform out of the box (asset management, multi-tenancy, maps, professional dashboards).
  • You're building solutions for clients/OEMs (white-labeling, scalable deployments).
  • You want built-in support for complex asset hierarchies and enterprise features.
  • You're working on smart buildings, energy, smart cities, or fleet-scale projects.

They Can Work Together​

Many people use Node-RED as a client/agent with OpenRemote (e.g., via MQTT) for heavy local processing or custom integrations. OpenRemote has a visual Flow editor that is somewhat inspired by tools like Node-RED but integrated into the broader platform.

Summary​

  • Node-RED = Powerful glue/tool for flows and integrations (very popular, flexible).
  • OpenRemote = Complete IoT platform for managing real-world deployments at scale.

If your project is small/personal → start with Node-RED.
If you're building a professional, maintainable IoT solution for multiple users/assets → OpenRemote (or Node-RED + a backend like Home Assistant/OpenHAB for simpler cases).

Both are excellent open-source options with no licensing costs. You can try Node-RED locally in minutes and OpenRemote via their Docker quickstart or demo.

Node-RED and OpenRemote integrate primarily through MQTT, making Node-RED an excellent edge/processing layer or gateway for OpenRemote's central IoT platform. This is the most common and well-documented pattern.

Primary Integration Method: MQTT​

OpenRemote includes a built-in MQTT broker (secured with TLS on port 8883 by default). Node-RED connects as an MQTT client to:

  • Publish sensor data, processed values, or commands into OpenRemote assets/attributes.
  • Subscribe to attribute changes, rules outputs, or commands from OpenRemote.

Setup Steps (High-Level)​

  1. In OpenRemote:

    • Create an Asset (e.g., Thing, Building, or custom type) with relevant attributes.
    • Create a Service User (in Users → Service User tab) with Read/Write roles. Copy the generated secret.
    • Link the service user to the asset (via User Asset Links) for fine-grained access control.
    • Note your realm name (usually master by default).

  2. In Node-RED:

    • Use the built-in MQTT In and MQTT Out nodes.
    • Broker configuration:
      • Server: Your OpenRemote hostname/IP.
      • Port: 8883 (TLS).
      • Protocol: MQTT v5 or v3.1.1.
      • Client ID: Unique per connection.
      • Username: realm:service_username (e.g., master:mqttuser).
      • Password: The service user secret.
      • TLS: Enable; often works with "verify server certificate" or self-signed adjustments depending on your setup.

  3. MQTT Topics (OpenRemote conventions):

    • Publishing attribute values typically uses structured topics like:
      • realm/assetId/attributeName or specific write paths.
    • OpenRemote automatically maps incoming MQTT messages to asset attributes when properly configured (via service user links and protected attributes).

Refer to OpenRemote's official MQTT documentation and wiki for exact topic formats and examples (e.g., Tutorial: Connect your MQTT Client).

Common Use Cases​

  • Edge Processing: Run Node-RED on Raspberry Pi, industrial gateways, or edge devices for local logic, protocol translation (Modbus, BACnet, serial, etc.), data filtering, or aggregation, then forward clean data to OpenRemote via MQTT.
  • Custom Integrations: Connect devices/protocols without native OpenRemote agents (e.g., Philips Hue, specific APIs, databases) using Node-RED's vast node library, then bridge to OpenRemote.
  • Complex Logic: Use Node-RED for advanced flows, JavaScript functions, machine learning, or external service calls that are cumbersome in OpenRemote's Flow Editor or rules.
  • Bidirectional Control: Node-RED subscribes to OpenRemote attribute changes (e.g., setpoints from dashboards) and sends commands back to hardware.
  • Data Enrichment: Pull from external sources (weather, databases, other systems) and push enriched data into OpenRemote.

Other Integration Possibilities​

  • REST API / HTTP: Node-RED's HTTP nodes can call OpenRemote's Manager APIs for assets, rules, users, etc. Useful for one-off operations or when MQTT isn't ideal.
  • OpenRemote as Backend + Node-RED Flows: Use OpenRemote for asset modeling, multi-tenancy, dashboards, and alarms; Node-RED for heavy lifting at the edge.
  • Node-RED as Agent Alternative: For scenarios where OpenRemote's built-in agents don't suffice, Node-RED + MQTT is a popular flexible substitute.
  • OpenRemote Flow Editor: OpenRemote has its own visual flow editor (somewhat inspired by Node-RED), but many users still prefer full Node-RED for complex edge logic.

Tips & Gotchas​

  • Security: Always use TLS and service users. Avoid exposing the broker publicly without proper authentication.
  • TLS Issues: Some users disable certificate verification initially for testing (not recommended for production). With proper domain certificates, standard verification usually works.
  • Debugging: Check OpenRemote Manager logs and Node-RED debug nodes. Use tools like MQTT Explorer for testing topics first.
  • Scalability: This combo works well — OpenRemote handles the platform layer (digital twins, rules, UI), while Node-RED excels at lightweight, distributed edge processing.
  • Community Examples: Search the OpenRemote forum for threads on Node-RED + MQTT; many users share configs for specific devices.

When This Integration Shines​

  • Hybrid Deployments: Central OpenRemote instance + multiple Node-RED edge instances.
  • Prototyping to Production: Quick experimentation in Node-RED, then structured management in OpenRemote.
  • Protocol Glue: Node-RED handles niche protocols and transforms data into OpenRemote's asset model.

Both tools are open-source and Docker-friendly, so you can run them side-by-side easily.

If you have a specific scenario (e.g., particular hardware, data flow direction, or edge device), provide more details and I can give tailored flow examples or configuration snippets. You can also try the official OpenRemote demo/quickstart alongside a local Node-RED instance to experiment.
